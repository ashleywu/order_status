# Milestone 1 — 需求冻结 + Airtable Base

与总计划对齐：[plan-airtable.md](plan-airtable.md) · [request.md](request.md)

**目标**：在写任何采集器代码之前，把 **数据真相（Airtable）** 建好，字段名固定、可做独立验收。

**原则**：所有字段 **Field name（API）** 使用 **`snake_case` 英文**，与 `plan-airtable.md` §2 一致。

**仓库已实现（辅助落地）**：见 [airtable/README.md](airtable/README.md) · 机器可读 [airtable/schema-m1.json](airtable/schema-m1.json) · 示例 CSV [airtable/seeds/](airtable/seeds/)。

---

## 1. 需求冻结（M1 开始前完成）

在下面签字/打勾后即视为冻结本里程碑范围（细节仍可在 Base 内调优，不改 API 字段名语义）：

- [ ] **单位约定**：Ingredient 类主要用何种 `unit`（如 `ml` / `g`，精油按 **`drops`（滴）** 也行）；瓶贴丝带包装等主要用 `piece` / `sheet` / `roll` 等，并写入团队备忘。
- [ ] **`category`** 五选一已确认：`ingredient` / `bottle` / `label` / `ribbon` / `packaging`。
- [ ] **`material_group`**（Single select）：至少列出首期选项（例如：常用精油、基底油、冷门精油、瓶子、标签、包装…），可按业务增删选项，**不改变 `material_group` API 字段名**。
- [ ] **高频物料**：哪些打 `favorite = true`，初版 **`sort_order`** 排序意向（小号靠前或按约定规则）。

产出：一段话或一页表（可到 Notion/Sheet），与 Base 示例数据一致即可。

## 2. 建 Airtable Base

- [ ] 新建 Base（名称自定，例如 `Lab Consumption`）。
- [ ] Base 内建 **4 张表**（表名可与下表脚本一致，方便 env：`Suppliers`、`Materials`、`Material_batches`、`Consumption_logs`，或统一前缀由你自定，但 **API table id / 英文名**需在后续填入 Vercel）。

## 各表 Primary field（Airtable）

每张表必须指定 **Primary field**（首列）；下列为 M1 **冻结约定**，避免用手机端会大量用到的 link/日期做主列而导致手工建表别扭。

| 表 | Primary field（API） | 说明 |
|----|----------------------|------|
| `Suppliers` | **`name`** | Single line text |
| `Materials` | **`name`** | Single line text |
| `Material_batches` | **`batch_code`** | Single line text；采购批次可读主键 |
| `Consumption_logs` | **`log_id`** | **Autonumber** 或 **Formula**（生成稳定可读标识即可）；业务列里的 `material` / `occurred_at` **不要**顶替 Primary |

**Consumption_logs**：除主列外可加 **`created_at`**（字段类型 **Created time**）仅供审计——**不可替代**下面的业务字段 **`occurred_at`**。

## 4. `Materials`（手机选料核心）

## 6. `Consumption_logs`（瘦表）

**Primary field：`log_id`**（Autonumber 或 Formula，见上文「各表 Primary field」）

| Field name（API） | 类型 | 备注 |
|-------------------|------|------|
| `log_id` | Autonumber 或 Formula | **Primary field** |
| `material` | Link → Materials | 必填逻辑由录入规则保证 |
| `usage_type` | Single select | `rd_lab`, `product`, `marketing`, `workshop` |
| `quantity` | Number | 消耗记 **正数** |
| `occurred_at` | **Date with time** | 业务发生时间；**服务端写入时可填当前时间** |
| `operator` | Single line text | 可选 |
| `device_id` | Single line text | 可选 |
| `client_request_id` | Single line text | 幂待用，可先空或填测试 UUID |
| `voided` | Checkbox | |
| `void_reason` | Long text | 可选 |
| `note` | Long text | 可选 |
| `created_at` | Created time | **可选**；仅审计，不可替代 `occurred_at` |

**禁止**在本表冗余：`supplier`、`color`、`image`、单价、批次细节等——一律通过 `material` 或日后 lookup 视图查看。

- [ ] 表已创建；
- [ ] **无**「supplier」「color」「image」等镜像列（若在 Airtable 里用 **Lookup** 做报表视图，仍可接受，不作为「写入冗余」）。

## 7. Airtable 视图（建议）

- [ ] **Materials**：`Active materials`（`active = true`）、`Favorites`、`By category`、`By material_group`、`Inactive`
- [ ] **Material_batches**：`Recent purchases`、`By supplier`、`By material`
- [ ] **Consumption_logs**：`Today`、`This week`、`This month`、`By usage type`、`Voided`

视图名称可本地化，核心是 **过滤器/分组** 能支撑运营习惯。

## 8. 示例数据（M1 必做）

至少满足：

- [ ] **≥2 条 Suppliers**。
- [ ] **≥10 条 Materials**，覆盖 **多种 `category` 与 `material_group`**。
- [ ] **`favorite`、`sort_order`、`active`** 有明显差异（便于验收「谁在首页」「谁不出现」）。
- [ ] **≥2 条 Material_batches**，挂到不同 `material`。
- [ ] **≥3 条 Consumption_logs**：含不同 `usage_type`，至少一行 `voided = true` 用于视图验收（可先手工填）。
- [ ] **`client_request_id`** 至少在两条测试记录上填入 **不同** 的假 UUID（为后续 §5.2 幂等公式预埋习惯）。

（仓库提供 `airtable/seeds/*.csv` 可直接导入或使用 README 顺序手工对齐。）
